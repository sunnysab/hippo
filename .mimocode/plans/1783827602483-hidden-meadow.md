# Hippo Sync 优化计划

## 问题分析

### 1. 增量 vs 全量同步逻辑是否重复？

**结论：有概念重叠，但有实际差异，不建议完全合并。**

| 维度 | 全量同步 (full) | 增量同步 (incremental) |
|------|----------------|----------------------|
| 停止条件 | 整页文章都已存在 | 遇到 `last_synced_at` 之前的文章 |
| 断点续传 | 有 (`resume_key`) | 无 |
| 跳过逻辑 | `completed_today` (同一天不重跑) | `sync_interval` + `skip_minutes` |
| 时间过滤 | 无 | 有 (`since_timestamp`) |
| 适用场景 | 首次全量拉取、历史数据补全 | 日常增量更新 |

**核心差异**：全量同步的断点续传依赖 `resume_key`，但这个 key 在非 bulk 模式下会被禁用（line 239-240）。增量同步通过时间戳自然停止，更适合日常使用。

**如果要简化**：可以将增量同步视为"时间窗口版的全量同步"，但保留两种模式的 API 接口，内部统一处理逻辑。

### 2. 失败文章重试问题

**问题根因**：`_select_missing_content()` 每次都查询所有没有内容的文章，没有任何重试次数限制。

```python
# sync_service.py line 121-137
def _select_missing_content(storage, biz, *, limit):
    articles = storage.articles.list_articles(biz, limit=limit)  # 无 exclude_downloaded
    ids = storage.articles.get_article_content_ids(biz, ...)
    return [a for a in articles if a.article_id not in ids]  # 返回所有无内容文章
```

**问题**：
- `list_articles` 没有 `exclude_downloaded=True`，返回所有文章
- 没有重试次数限制，失败 100 次的文章也会被重试
- `limit=None` 时（bulk 模式），返回全部无内容文章

**建议方案**：在 `article_content` 表添加下载追踪字段，实现按文章的重试上限。

## 实施计划

### Phase 1: Schema 变更 - 新增独立追踪表

**文件**: `schema/postgres.sql`

```sql
CREATE TABLE IF NOT EXISTS article_download_attempts (
    id SERIAL PRIMARY KEY,
    biz TEXT NOT NULL REFERENCES accounts(biz) ON DELETE CASCADE,
    article_id TEXT NOT NULL,
    attempts INTEGER NOT NULL DEFAULT 0,
    last_error TEXT,
    last_attempt_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL,
    UNIQUE (biz, article_id)
);

CREATE INDEX IF NOT EXISTS idx_article_download_attempts_biz_article
ON article_download_attempts (biz, article_id);
```

**文件**: `hippo/storage.py` (SCHEMA_VERSION 从 '19' 升级到 '20')

### Phase 2: Repository 层更新

**文件**: `hippo/repositories.py`

新增 `DownloadAttemptRepository` 类：

```python
class DownloadAttemptRepository:
    def __init__(self, conn):
        self._conn = conn
    
    def get_attempts(self, biz: str, article_id: str) -> int:
        """获取文章下载尝试次数"""
        
    def get_attempts_map(self, biz: str, article_ids: Iterable[str]) -> dict[str, int]:
        """批量获取下载尝试次数"""
        
    def increment_attempt(self, biz: str, article_id: str, error: str | None = None) -> None:
        """递增尝试次数，更新 last_error 和 last_attempt_at"""
        
    def reset_attempts(self, biz: str, article_id: str) -> None:
        """重置尝试次数（可选，用于成功后重置）"""
        
    def get_articles_within_limit(
        self, biz: str, article_ids: Iterable[str], *, max_attempts: int
    ) -> set[str]:
        """返回尝试次数未超限的文章 ID 集合"""
```

### Phase 4: Sync Service 更新

**文件**: `hippo/sync_service.py`

修改 `_select_missing_content` - 使用新的追踪表过滤：

```python
def _select_missing_content(
    storage: PostgresStorage,
    biz: str,
    *,
    limit: int | None,
    max_download_attempts: int = 3,  # 新增参数，默认 3 次
) -> list[ArticleRecord]:
    if limit is not None and limit <= 0:
        return []
    
    # 获取所有无内容的文章
    articles = storage.articles.list_articles(biz, limit=limit)
    ids = storage.articles.get_article_content_ids(biz, [a.article_id for a in articles])
    missing = [a for a in articles if a.article_id not in ids]
    
    if not missing:
        return []
    
    # 新增：过滤掉尝试次数超限的文章
    if max_download_attempts is not None:
        missing_ids = [a.article_id for a in missing]
        within_limit = storage.download_attempts.get_articles_within_limit(
            biz, missing_ids, max_attempts=max_download_attempts
        )
        missing = [a for a in missing if a.article_id in within_limit]
    
    return missing
```

修改 `sync_accounts` 传入配置：

## 回答你的三个问题

### Q1: 增量和全量逻辑是否重复？

**有概念重叠，但有实际差异**：
- 全量同步有断点续传（`resume_key`），增量没有
- 增量通过时间戳自然停止，更高效
- 全量有 `completed_today` 优化
- 建议保持两种模式，但可以统一底层实现逻辑

### Q2: 是否应该给失败文章加重试上限？

**建议采纳**：
- 新增独立的 `article_download_attempts` 表追踪下载尝试
- 默认上限 3 次（可配置）
- 超过上限的文章不再被 `_select_missing_content` 返回
- 这样可以避免每次都重试大量历史失败文章

### Q3: Review 你的计划

**你的观察很准确**：
- 138/256 的数量确实可能包含大量历史失败文章的重试
- 失败文章应该有上限，否则每次同步都会重复尝试
- 增量/全量可以保持现状，但可以优化底层实现
